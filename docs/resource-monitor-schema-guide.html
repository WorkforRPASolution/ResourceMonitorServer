설비 모니터링 기준정보 — SCHEMA 완전 해설

컬렉션 RESOURCE_MONITOR_PROFILE 의 모든 항목과 설정 가능한 값을, 코딩을 모르는 설비 엔지니어가 읽고 이해할 수 있게 하나도 빠짐없이 설명합니다.

이 문서는 읽고 이해하는 레퍼런스입니다. 값을 직접 만들어 보고 싶다면 설정 Playground를 함께 쓰세요.

근거: SCHEMA.md v2.0 (1차 진실) 메트릭: EARS-METRICS-REFERENCE.md 검증 동작: src/db/models.py · fact_catalog.py Phase 1 구현완료 · Phase 2 · Phase 3

§1 큰 그림 — 한 장비 설정은 4계층 SCHEMA §0.2 / §1.1

모니터링 설정은 컬렉션 하나(RESOURCE_MONITOR_PROFILE)로 관리합니다. "단순 임계값"과 "복합 조건"을 따로 두지 않습니다 — 둘 다 같은 구조이고, 조건 개수만 다릅니다(단순=조건 1개, 복합=조건 여러 개). 한 문서(프로파일) 안에 네 가지 계층이 들어갑니다.

계층역할비유주요 내용
scope이 설정을 어느 장비에 적용할지🎯 누구에게process / eqpModel / eqpId
measures[]무엇을 어떻게 잴지 (집계 → fact 산출)🌡️ 잰다category·metric·window + facts
rules[]잰 값으로 언제 경보할지 (조건 + 평가 주기)🚨 판단한다when 조건 + interval + severity
notify{}어떻게 알릴지 (이메일·억제)📧 알린다cooldown + email 코드

데이터 흐름

scope로 대상 결정 → measures로 잼 → rules로 판단 → notify로 발송. 이 순서가 전체 시스템의 한 사이클입니다.

한 문서의 뼈대

// RESOURCE_MONITOR_PROFILE 문서 1개
{
  "scope":    { "process": "*", "eqpModel": "*", "eqpId": "*" },  // 누구에게
  "enabled":  true,                                              // 켜고 끄기
  "governance": { "version": 1, "updated_by": "..." },        // 변경 이력/낙관락
  "measures": [ /* 잰다: 무엇을·어떻게 → fact 산출 */ ],
  "rules":    [ /* 판단: fact로 경보 결정 + 평가 주기 */ ],
  "notify":   { /* 알린다: 전달 채널 */ }
}

enabled 는 이 설정 전체를 켜고 끄는 스위치(기본 true)이고, governance 는 누가 언제 왜 바꿨는지(version·updated_by·updated_at·change_reason)를 담는 변경 이력입니다. version 은 동시 수정 충돌을 막는 낙관적 잠금 토큰으로도 쓰입니다. (이 문서 전체 스위치와 별개로, rule 하나만 끄는 rule.enabled도 있습니다 — §6 예시 ⑨.)

DB 위치 모든 컬렉션은 단일 EARS MongoDB에 공존합니다. RMS는 RESOURCE_MONITOR_PROFILE 만 소유(읽기·쓰기)하고 EQP_INFO 등은 read-only로 참조합니다.

§2 scope — 어느 장비에 적용할까 SCHEMA §1.2

🎯 누구에게 — 이 설정이 적용될 장비 범위를 정합니다. EQP_INFO(장비 마스터)의 계층을 그대로 따릅니다. 넓은 범위(전체)부터 좁은 범위(특정 장비 1대)까지 3단계입니다.

경로타입필수기본의미
scope.processstring✓—EQP_INFO.process. "*"=전체. 파티션/인덱스 라우팅 키
scope.eqpModelstring선택"*"장비 모델. 카멜케이스
scope.eqpIdstring선택"*"단일 장비 1대

이름 표기 주의 — eqpModel / model / eqp_model

같은 항목이 위치마다 이름이 다릅니다. 모두 같은 값입니다.

위치이름
JSON으로 보낼 때(API)model
MongoDB에 저장된 키eqpModel (카멜케이스)
파이썬 내부 코드eqp_model (Pydantic alias)

마찬가지로 단일 장비는 JSON·Mongo에서 eqpId, 파이썬에서 eqp_id 입니다. "*" 는 "해당 단계 전체"를 뜻하는 와일드카드입니다.

scope 작성 예시

// 전역: 모든 장비에 적용
"scope": { "process": "*", "model": "*", "eqpId": "*" }

// 특정 공정 전체
"scope": { "process": "PHOTO", "model": "*", "eqpId": "*" }

// 단일 장비 1대 (가장 구체적)
"scope": { "process": "PHOTO", "model": "MODEL_A", "eqpId": "EQP001" }

읽는 법: 위로 갈수록 넓고 아래로 갈수록 좁습니다. 좁은 문서가 넓은 문서를 상속하고 바꿀 것만 덮습니다(§10).

⚠️ 혼동 금지 scope.process(설정 적용 범위 = ES 인덱스 파티션 키)와 measure의 proc(EARS 행의 정체성 = EARS_PROCNAME)은 완전히 다른 개념입니다. 이름이 비슷할 뿐입니다(함정 P2).

구체적인 scope 문서는 더 넓은 scope를 상속하고 "바꿀 것만" 담습니다(전체 복사 금지). 합성 규칙은 §10 scope 상속에서 자세히 다룹니다.

§3 measures — 무엇을 어떻게 잴까 SCHEMA §1.3

🌡️ 잰다 — 각 measure는 "무엇을(어떤 메트릭) 어떻게(어떤 집계로) 재서 어떤 fact(잰 결과값)를 만들지"의 정의입니다. measure는 집계창(window)만 갖고, 평가 주기(interval)는 갖지 않습니다 — 주기는 rule의 몫입니다.

measure의 모든 필드

경로타입필수기본의미
idstring✓—문서 내 유일. rule이 "id.type"으로 참조하는 손잡이
categorystring✓—EARS category(cpu/memory/disk…). 메트릭 혼입 방지에 필수
metricstring✓—EARS metric. 와일드카드 가능("*_core_load", "*")
procstring선택"@system"EARS proc. "*"이면 proc별로 fact 산출
window_minutesint✓—집계 시간창(분). 이 measure의 모든 fact가 공유
group_bystring[]선택["eqpId"]묶음 기준. proc=="*"이면 자동 ["eqpId","proc"]
expandenum선택"scalar"scalar/instance. metric이 와일드카드면 자동 instance
metric_kindenum｜null선택nullgauge/counter/cumulative. lint 힌트(강제 아님)
bucketingobject｜null조건부null시간축 fact가 있으면 필수
baselineobject｜null조건부nullbaseline_dev fact가 있으면 필수
facts[]array✓—산출할 fact 목록. 항목 1개 = fact 1개(§4)

group_by 와 expand 의 값

  • group_by 허용 값: "eqpId", "proc". 보통 장비별(eqpId)로 묶고, proc별로도 나누려면 proc을 추가합니다(proc=="*"이면 자동 추가).
  • expand = "scalar" 는 메트릭 하나당 fact 하나. "instance" 는 와일드카드/다중 인스턴스(디스크 C:·D:, 센서 여러 개 등)마다 fact 집합을 산출합니다. 이때 rule에서 정량자(quantifier)로 "하나라도/모두/N개 이상"을 정합니다.
  • metric_kind 는 강제 규칙이 아니라 힌트입니다. gauge(현재값: CPU%·온도), counter(누적 증가: 에러 수), cumulative. gauge에 delta/growth_rate를 쓰면 lint 경고가 납니다(검증 10).

bucketing — 시간축 fact용 잘게 쪼갠 창

duration·growth_rate·moving_avg·trend 같은 "시간에 따른 변화"를 보는 fact는, 큰 창을 작은 버킷으로 쪼개 계산합니다. 그 버킷 정의가 bucketing 입니다.

경로타입필수기본의미
bucketing.secondsint✓*—버킷 하나의 크기(초). date_histogram 간격
bucketing.pointsint｜null조건부nullmoving_avg/trend가 있으면 필수(평활화 점 개수)

* bucketing 객체를 두면 seconds는 반드시 있어야 합니다. 제약: seconds × points ≤ window_minutes × 60(검증 3).

baseline — 과거 대비용 기준선 Phase 3

baseline_dev fact(과거 동일 시간대 대비 편차%)가 있을 때만 필요합니다.

경로타입기본의미
baseline.daysint7과거 며칠을 기준선으로 볼지
baseline.same_hourbooltrue동일 시간대만 비교(예: 매일 14시끼리)
baseline.min_pointsint30표본이 이보다 적으면 경보하지 않음
baseline.deviation_floorfloat1.0분모≈0일 때 편차% 폭주를 막는 하한
핵심 원칙 1 measure 항목 = 1 fact = 1 type. bucketing·baseline처럼 여러 통계가 함께 쓰는 파라미터는 measure 레벨에 한 번만 둬서 일관성을 강제합니다(예: moving_avg와 trend가 같은 평활화 창을 쓰도록).

measure 작성 예시

① 단순 스칼라 — CPU 사용률 한 종류, fact 여러 개

{ "id": "cpu", "category": "cpu", "metric": "total_used_pct", "proc": "@system",
  "window_minutes": 15, "metric_kind": "gauge",
  "facts": [ { "type": "max" }, { "type": "avg" }, { "type": "p95" } ] }

읽는 법: 15분 창에서 cpu의 total_used_pct를 재고, 그 창의 최댓값·평균·95백분위 세 fact를 산출. rule은 cpu.max·cpu.avg·cpu.p95로 부릅니다.

② 인스턴스 확장 — 와일드카드 metric(디스크 파티션 전부)

{ "id": "disk", "category": "disk", "metric": "*", "proc": "@system",
  "window_minutes": 30, "expand": "instance",
  "facts": [ { "type": "max" } ] }

읽는 법: metric:"*"이라 C:·D: 등 파티션마다 max를 산출(expand는 자동으로 instance). rule에서 정량자로 "하나라도/모두"를 정합니다.

③ 시간축 — bucketing 필요(증가율·추세)

{ "id": "mem_size", "category": "memory", "metric": "total_used_size", "proc": "@system",
  "window_minutes": 60, "metric_kind": "gauge",
  "bucketing": { "seconds": 300, "points": 6 },
  "facts": [ { "type": "growth_rate", "unit": "per_hour" }, { "type": "trend" } ] }

읽는 법: growth_rate·trend는 시간축 fact라 bucketing 필수, trend 때문에 points도 필수(검증 2). 300×6=1800초 ≤ 60분×60이라 검증 3 통과.

§4 fact 카탈로그 — 16종 전수 SCHEMA §2 / fact_catalog.py

fact = measure가 만들어내는 잰 결과값입니다. facts[].type 에 넣는 값이 곧 fact 이름이고, rule에서 measureId.type(예: cpu.p95)으로 부릅니다. 닫힌 목록이라 자유 텍스트는 쓸 수 없습니다. 아래 16종이 전부입니다.

Phase 1 fact — 지금 엔진이 동작 구현완료

max · 창 내 최댓값

필수 params: 없음
허용 op: > >= (높을 때만)
value 단위: 메트릭 단위
window 따름: ✅
예: cpu.max >= 95 → 창 안에서 CPU가 한 번이라도 95% 이상이면 경보.

min · 창 내 최솟값

필수 params: 없음
허용 op: < <= ==
value 단위: 메트릭 단위
window 따름: ✅
예: mem_free.min <= 5(여유 메모리 바닥), proc_required.min == 0(필수 프로세스 다운). ==가 허용되는 건 상태 점검을 위해서입니다.

avg · 창 내 평균

필수 params: 없음
허용 op: > >= < <=
value 단위: 메트릭 단위
window 따름: ✅
예: cpu.avg >= 70 → 평균적으로 부하가 높은 상태.

last · 창 내 마지막(최신) 값

필수 params: 없음
허용 op: > >= < <= == !=
value 단위: 메트릭 단위
window 따름: ✅
가장 최근 샘플 1개를 봅니다. ES top_hits를 쓰므로 비쌉니다 — 꼭 필요할 때만.

p50 p90 p95 p99 · 백분위수

필수 params: 없음
허용 op: > >= < <=
value 단위: 메트릭 단위
window 따름: ✅
p95=상위 5%를 제외한 값. 순간 튐(spike)에 둔감하고 "대체로 높은" 상태를 잡습니다. 예: cpu.p95 > 80.

spike_count · 사건 횟수

필수 params: over, direction
허용 op: > >=
value 단위: 정수(횟수)
window 따름: ✅
over(경계값)를 direction(above/below) 방향으로 넘긴 샘플 수. 예: over:90, direction:above + rule spike_count > 5 → 90% 초과가 5번 넘게 발생.

Phase 2 fact — 스키마·검증은 받지만 엔진은 아직 skip+경고 후속

duration · 최대 연속 지속(초)

필수 params: over, direction + measure bucketing.seconds
허용 op: > >=
value 단위: 초
window 따름: ✅
over 경계를 연속으로 넘긴 가장 긴 시간(초). 예: 80% 초과가 > 180초 지속.

delta · 변화량

필수 params: mode(last_minus_first/max_minus_min)
허용 op: > >= < <= !=
value 단위: 메트릭 단위
window 따름: ✅
창 안에서의 변화량. 누적 카운터(에러 수 등)에 적합. 예: ssd_err.delta > 0 → 새 미디어 에러 발생.

growth_rate · 단위시간당 증가

필수 params: unit(per_hour/per_day) + measure bucketing.seconds
허용 op: > >= < <=
value 단위: 단위/시간
window 따름: ✅
기울기(추세선). 예: 메모리가 시간당 50MB 이상 증가 → 누수 의심.

moving_avg · 이동평균값

필수 params: measure bucketing.seconds + points
허용 op: > >= < <=
value 단위: 메트릭 단위
window 따름: ✅
points개 버킷으로 평활화한 평균. 노이즈를 줄여 추세를 봅니다.

trend · 추세 방향

필수 params: measure bucketing.seconds + points
허용 op: trend== 만
value 단위: enum(문자열) 예 "increasing"
window 따름: ✅
유일하게 op가 trend==이고 value가 문자열입니다. 예: mem_size.trend trend== "increasing".

zscore · 표준화 이상도(창 내부)

필수 params: direction(high/low, 기본 high)
허용 op: > >=
value 단위: σ(표준편차)
window 따름: ✅
(값 − 평균) / 표준편차. 창 내부 분포에서 얼마나 튀는지. spike_count·duration의 above/below와 달리 zscore의 direction은 high/low입니다.

Phase 3 fact — 과거 인덱스 기반 후속

baseline_dev · 과거 동일시간대 대비 편차%

필수 params: measure baseline{days, same_hour…}
허용 op: > >= < <=
value 단위: %
window 따름: ❌ (과거 N일 인덱스를 별도 쿼리)
"평소(과거 같은 시간대)보다 몇 % 벗어났나". 유일하게 현재 창이 아니라 과거 인덱스를 따로 조회합니다.

§5 fact 조건부 파라미터 — over·direction·mode·unit SCHEMA §1.3 / models.Fact

facts[] 항목은 type 외에 type에 따라 필요한 추가 파라미터를 가질 수 있습니다. 필요 없는 type에 넣으면 거부됩니다(extra="forbid").

파라미터타입/값어떤 type에의미
overfloatspike_count, duration (필수)"사건"으로 셀 경계값(예: 90%)
directionenumspike_count·duration: above/below · zscore: high/low경계를 어느 방향으로 넘겼는지
modeenumdelta (기본 last_minus_first)변화량 계산법: last_minus_first/max_minus_min
unitenumgrowth_rate (기본 per_hour)증가율 기준: per_hour/per_day
⚠️ direction 값이 type마다 다릅니다 spike_count·duration은 above/below, zscore는 high/low. 섞으면 거부됩니다.
// 조건부 파라미터 예시
{ "type": "spike_count", "over": 90, "direction": "above" }
{ "type": "duration",    "over": 80, "direction": "above" }
{ "type": "delta",       "mode": "last_minus_first" }
{ "type": "growth_rate", "unit": "per_hour" }
{ "type": "zscore",      "direction": "high" }

§6 rules — 언제 경보할까 + 평가 주기 SCHEMA §1.4

🚨 판단한다 — rule은 measure가 만든 fact를 보고 "경보할지"를 정합니다. measure가 창(window)을 가졌다면, rule은 주기(interval)를 가집니다 — "얼마나 자주 판단하나".

경로타입필수기본의미
idstring✓—문서 내 유일
interval_minutesint✓—평가 주기(분). 스케줄 단위 = (process × rule)
severityenum✓—WARNING / CRITICAL
combineenum선택AND여러 조건 결합: AND / OR
when[]array✓—조건 목록(최소 1개)
when[].factstring✓—"measureId.type"(예: "cpu.p95")
when[].openum✓—>= > <= < == != trend==
when[].valuefloat｜string✓—비교 기준값. trend==일 때만 문자열
when[].quantifierenum선택any인스턴스 정량화: any/all/count
when[].count_minint｜null조건부nullquantifier=="count"일 때 필수(≥1)
notifystring선택"default"notify 맵의 채널 이름
enabledbool선택true이 rule 켜고 끄기. false면 엔진이 평가 스킵 + 스케줄 cadence에서 제외(profile.enabled와 별개)

when[].fact 의 점 표기

"cpu.p95" = 점 앞은 measure id, 점 뒤는 그 measure가 선언한 fact type. 둘 다 실제로 존재해야 저장됩니다(검증 5).

2단계 임계는 rule 2개로 WARNING 80 / CRITICAL 95 같은 2단계 경보는 같은 measure를 참조하는 rule 2개(cpu_warn, cpu_crit)로 표현합니다. measure를 늘리지 않는 정석입니다.

combine 으로 단순/복합 통일

  • 조건 1개 = 단순 임계값. 조건 여러 개 + combine = 복합 조건.
  • AND: 모든 조건이 동시에 참이어야 경보(예: p95>80 그리고 spike_count>5 그리고 duration>180).
  • OR: 하나라도 참이면 경보. 범위 이탈(전압 등)은 단일 연산자가 아니라 두 조건 + combine:"OR" 입니다: min < 하한 OR max > 상한.

when 절 — 패턴별 작성 예시

아래 예시는 모두 §14·SCHEMA §12에 나오는 measure(cpu·mem_free·disk·volt_vcore·proc_required·proc_forbidden·disk_health·mem_size)를 참조한다고 가정합니다.

① 단순 임계 — 조건 1개

{ "id": "cpu_warn", "interval_minutes": 5, "severity": "WARNING", "notify": "default",
  "when": [ { "fact": "cpu.max", "op": ">=", "value": 80 } ] }

읽는 법: cpu measure의 max fact가 80 이상이면 경보. 조건이 1개면 combine은 영향 없음.

② 2단계 임계(WARNING/CRITICAL) — 같은 fact, rule 2개

{ "id": "cpu_warn", "interval_minutes": 5, "severity": "WARNING", "notify": "default",
  "when": [ { "fact": "cpu.max", "op": ">=", "value": 80 } ] },
{ "id": "cpu_crit", "interval_minutes": 5, "severity": "CRITICAL", "notify": "default",
  "when": [ { "fact": "cpu.max", "op": ">=", "value": 95 } ] }

읽는 법: 임계 2단계는 measure를 늘리지 않고 rule 2개로. 80~95는 WARNING, 95↑는 CRITICAL.

③ 복합 AND — 세 조건이 모두 참일 때

{ "id": "cpu_anomaly", "interval_minutes": 5, "severity": "CRITICAL", "combine": "AND", "notify": "default",
  "when": [
    { "fact": "cpu.p95",         "op": ">", "value": 80 },
    { "fact": "cpu.spike_count", "op": ">", "value": 5 },
    { "fact": "cpu.duration",    "op": ">", "value": 180 }
  ] }

읽는 법: "대체로 높고(p95>80) 그리고 튐이 잦고(5회 초과) 그리고 오래 지속(180초 초과)". 세 조건이 같은 cpu measure(같은 proc)를 참조하므로 검증 8 통과.

④ 범위 이탈 OR — 하한 미만 또는 상한 초과 (전압)

{ "id": "volt_out", "interval_minutes": 5, "severity": "WARNING", "combine": "OR", "notify": "default",
  "when": [
    { "fact": "volt_vcore.min", "op": "<", "value": 1.1 },
    { "fact": "volt_vcore.max", "op": ">", "value": 1.4 }
  ] }

읽는 법: 정상 범위 1.1~1.4V를 벗어나면 경보. 범위 이탈은 단일 연산자로 못 쓰고 두 조건 OR로 표현합니다.

⑤ 정량자 — any / all / count (인스턴스 measure)

// any: 디스크 파티션 중 하나라도 95%↑
{ "fact": "disk.max", "op": ">=", "value": 95, "quantifier": "any" }
// all: 모든 파티션이 90%↑ 일 때만
{ "fact": "disk.max", "op": ">=", "value": 90, "quantifier": "all" }
// count: 온도 센서 3개 이상이 90℃↑ (count_min 필수)
{ "fact": "temp.max", "op": ">=", "value": 90, "quantifier": "count", "count_min": 3 }

읽는 법: expand:"instance" measure는 인스턴스(C:·D:, 센서 여러 개)마다 fact를 만듭니다. 그 집합을 "하나라도/모두/N개 이상"으로 묶는 게 정량자입니다.

⑥ 상태 점검 — min/max로 표현 (별도 type 없음)

// 필수 프로세스 다운
{ "fact": "proc_required.min",  "op": "==", "value": 0, "quantifier": "any" }
// 금지 프로세스 실행
{ "fact": "proc_forbidden.max", "op": ">",  "value": 0, "quantifier": "any" }
// 디스크 건강 위험(PRED_FAIL=2 이상)
{ "fact": "disk_health.max",   "op": ">=", "value": 2, "quantifier": "any" }

읽는 법: 실행 여부(1/0)·상태값을 min/max + ==/>/>=로 점검합니다(§4 상태 점검).

⑦ 추세 — trend== + 보조 조건 AND (메모리 누수)

{ "id": "mem_leak", "interval_minutes": 5, "severity": "CRITICAL", "combine": "AND", "notify": "default",
  "when": [
    { "fact": "mem_size.trend",       "op": "trend==", "value": "increasing" },
    { "fact": "mem_size.growth_rate", "op": ">",       "value": 52428800 }
  ] }

읽는 법: trend==만 value가 문자열("increasing"). "증가 추세 그리고 시간당 50MB(52428800 byte) 초과 증가"면 누수로 판단.

⑧ AND·OR 혼합 — 한 rule로는 불가, rule을 나눠서 표현

❌ 한 rule 안에서 섞기 불가 combine은 rule 하나당 값 하나(AND 또는 OR)이고 그 rule의 모든 조건에 똑같이 적용됩니다. 조건별 그룹·괄호·중첩이 없으므로 (A AND B) OR C 같은 혼합식은 rule 하나로 못 만듭니다.

대신 식을 "AND 묶음들의 OR"(DNF)로 바꾼 뒤 AND 묶음 하나 = rule 하나로 나눕니다. 같은 notify 채널을 공유하는 rule들은 한 사건으로 묶이므로 rule 간 OR처럼 동작합니다.

// 원하는 논리: (cpu.p95 > 80  그리고  cpu.spike_count > 5)  또는  (cpu.avg >= 95)
{ "id": "cpu_anomaly", "interval_minutes": 5, "severity": "CRITICAL", "combine": "AND", "notify": "default",
  "when": [
    { "fact": "cpu.p95",         "op": ">", "value": 80 },
    { "fact": "cpu.spike_count", "op": ">", "value": 5 }
  ] },                                          // ← AND 묶음 = rule A
{ "id": "cpu_avg_crit", "interval_minutes": 5, "severity": "CRITICAL", "notify": "default",
  "when": [ { "fact": "cpu.avg", "op": ">=", "value": 95 } ] }  // ← 단독 조건 = rule B

읽는 법: rule A가 참이거나 rule B가 참이면 경보. A의 두 조건은 AND로, A·B 사이는 rule 분리로 OR. (A OR B) AND C처럼 OR이 안쪽이어도 분배법칙으로 (A AND C) OR (B AND C)로 풀어 rule 2개로 만듭니다.

⚠️ 반대 방향은 불가 이 방식은 OR(어느 묶음이라도 참 → 경보)에만 통합니다. "여러 rule이 동시에 참일 때만 경보"하는 rule 간 AND는 표현할 수 없습니다 — rule은 각자 따로 발화하기 때문입니다. 동시 충족이 필요하면 그 조건들을 한 rule에 combine:"AND"로 넣으세요(단, 모든 조건이 같은 proc 차원 measure 참조 — 검증 8).

⑨ rule 끄고 켜기 — enabled (개별 on/off)

rule 하나만 잠깐 끄고 싶을 때(시끄러운 경보 뮤트·점검 시간대·새 rule을 비활성으로 먼저 등록) enabled:false를 씁니다. profile 전체 스위치(profile.enabled)와 별개로 rule마다 적용됩니다. 끈 rule은 저장·검증은 그대로 되지만 엔진이 평가하지 않고 스케줄에서도 빠집니다.

// 이 rule만 비활성 — 저장은 되지만 발화하지 않음
{ "id": "cpu_warn", "interval_minutes": 5, "severity": "WARNING", "enabled": false, "notify": "default",
  "when": [ { "fact": "cpu.max", "op": ">=", "value": 80 } ] }

읽는 법: 기본은 true(생략 가능). 끄려면 "enabled": false. 다시 켤 때 안전하도록 끈 rule도 참조 무결성은 그대로 검사합니다(§9).

특정 장비에서만 상속 rule 끄기(소프트 tombstone) — eqp overlay에서 같은 id의 rule을 enabled:false로 다시 적으면, cascade가 rule을 통째로 교체하므로 그 장비에서만 해당 rule이 꺼집니다(전역은 그대로).

// (*,*,*) 전역은 cpu_warn 켜짐. 아래 overlay 문서만 추가하면 EQP001에서만 뮤트
{
  "scope": { "process": "PHOTO", "model": "MODEL_A", "eqpId": "EQP001" },
  "rules": [
    { "id": "cpu_warn", "interval_minutes": 5, "severity": "WARNING", "enabled": false,
      "when": [ { "fact": "cpu.max", "op": ">=", "value": 80 } ] }
  ]
}

읽는 법: 다른 장비는 전역 cpu_warn이 그대로 동작하고, EQP001만 이 rule이 꺼집니다. 상속 항목을 "통째로 삭제"하는 기능은 아직 없고(추후), 이 enabled:false 뮤트가 그 자리를 대신합니다(§10).

왜 measure가 window, rule이 interval을 갖나

측정의 속성(얼마나 긴 구간을 보나)은 measure에, 판단의 속성(얼마나 자주 보나)은 rule에 둡니다. 같은 measure를 주기가 같은 여러 rule이 참조하면 엔진이 한 번만 계산해 공유합니다. 스케줄 단위는 (process × interval) 묶음입니다.

§7 연산자(7) · 정량자(3) SCHEMA §3 / fact_catalog.ALLOWED_OPS

연산자 7종 when[].op

  • trend== 는 trend fact 전용이며 value가 문자열("increasing" 등)입니다.
  • 저장 시 type↔op 적합성을 검증합니다 — 예: max에 <=는 거부, trend는 trend==만 허용(검증 5).

정량자 3종 when[].quantifier

expand:"instance" measure(와일드카드 metric, 또는 proc별)는 장비마다 fact 집합을 만듭니다(디스크 C:/D:, 프로세스 여러 개 등). 그 집합을 "어떻게 묶어 판단할지"가 정량자입니다.

quantifier의미예
any (기본)인스턴스 중 하나라도 충족디스크 중 하나라도 95% 이상

count (+count_min)N개 이상 충족센서 3개 이상 과열(count_min:3)
⚠️ count_min quantifier=="count"면 count_min이 반드시 있어야 하고 1 이상이어야 합니다. count_min:0은 "0개 이상"이라 항상 참 → 무조건 알림이 되어 거부됩니다(검증 7).

§8 notify — 어떻게 알릴까 SCHEMA §1.5

📧 알린다 — "이름 → 채널" 맵입니다. rule은 채널 이름(notify 필드)으로 참조해 중복을 없앱니다. 보통 "default" 하나면 충분합니다.

경로타입필수기본의미
notify.<name>.cooldown_minutesint✓—같은 사건 재알림 억제 시간(Redis TTL)
notify.<name>.email_codestring선택"RESOURCE_MONITOR"EMAIL_TEMPLATE_REPOSITORY 매칭 코드
notify.<name>.email_subcodestring｜null선택nullnull이면 "{category}_{severity}" 자동

cooldown(이메일 폭주 방지) 키

같은 (장비, proc, 알림채널, 심각도) 단위로 재알림을 억제합니다. 키 형식:

{prefix}:cooldown:{process}:{eqpId}:{proc}:{notify}:{severity}

같은 notify 채널을 공유하는 rule들은 한 사건으로 묶여 함께 억제됩니다.

"notify": {

}

§9 저장 시 검증 규칙 — 10가지 SCHEMA §5 / validate_effective

프로파일을 저장할 때 아래 규칙을 검사하고, 위반하면 거부합니다(1~8). 9·10은 거부 대신 경고(로그)입니다. 검사 위치는 두 종류입니다 — 구조 검증(문서 한 장 단위, Pydantic 모델)과 참조·합성 검증(여러 scope를 합친 effective profile, 위반 시 HTTP 422).

#규칙무엇을 막나결과
1measure id 유일 / 한 measure 내 type 유일measureId.type 참조가 한 곳을 가리키도록(모호 참조 방지)거부
2시간축 fact(duration/growth_rate/moving_avg/trend) 있으면 bucketing 필수, moving_avg/trend엔 bucketing.points 필수버킷 정의 없이 시간축 계산 불가거부
3bucketing.seconds × points ≤ window_minutes × 60버킷이 창보다 커지는 모순거부
4baseline_dev 있으면 baseline 필수기준선 없이 편차% 계산 불가거부
5모든 rule.when.fact="M.T" → M 존재, T가 M의 facts에 선언됨, op가 T에 허용(ALLOWED_OPS)없는 measure/fact 참조, 부적합 연산자거부(422)
6rule.interval_minutes ≤ 참조 measure의 window_minutes판단 주기가 측정창보다 길어 사건 누락거부
7quantifier=="count"엔 count_min ≥ 1 필수 / rule.notify가 notify 맵에 존재무조건 알림(count_min 0), 없는 채널 참조거부(422)
8한 rule의 모든 조건은 동일 proc 차원 measure를 참조서로 다른 proc을 AND로 묶어 영영 발화 못 하는 silent lost breach거부(422)
9(lint) dead fact — 어떤 rule도 참조 않는 fact오타·잔재 발견(무해하나 점검)경고
10(lint) metric_kind=="gauge"인데 delta/growth_rate 사용변화량 fact를 gauge에 (보통 counter여야)경고
왜 합성 후 검증? 참조 무결성(5·6·7·8)은 overlay 한 장이 아니라 합성된 effective profile에 적용합니다. overlay 문서만 보면 cpu_crit이 참조하는 cpu measure가 없어 보여도, 전역과 합치면 존재하기 때문입니다(§10). 구조 검증(1~4)과 count_min≥1·값 타입은 문서 단위로 즉시 검사합니다.
비활성 rule도 검증 enabled:false인 rule도 위 검증을 그대로 받습니다(엄격). 끈 상태로 깨진 참조를 저장해 둘 수는 없으며, 그래서 다시 켜는 순간 항상 안전합니다. 단 interval-scope(§6.4)만은 활성 rule만 따집니다 — 비활성 rule이 새 주기를 들여와도 스케줄되지 않으므로 저장은 허용하고, 그 rule을 켤 때 검사합니다.

§10 scope 상속 — cascade(계층 합성) SCHEMA §6 / fold_profiles

한 장비의 유효 프로파일(effective profile)은 그 장비에 매칭되는 scope 문서들을 넓은 → 좁은 순으로 합성(fold)한 결과입니다. 구체적 scope는 더 넓은 scope를 상속하고 "바꿀 것만" 담습니다(= sparse overlay).

합성 순서 — 좁은 게 이김

(*,*,*)  →  (process,*,*)  →  (process,model,*)  →  (process,model,eqpId)

base에 overlay를 차례로 덮습니다. uniq_scope 인덱스 덕에 각 레벨 최대 1문서라 순서 모호성이 없습니다. 현재 운영은 대부분 전역 + 가장 구체적 1개의 2-레이어로 충족됩니다(YAGNI).

합성 규칙 — key 기준 병합

대상병합 키충돌 시
measuresmeasure.id구체 문서의 measure가 통째로 교체
rulesrule.id구체 문서의 rule이 통째로 교체
notify맵 이름구체 문서의 채널이 통째로 교체
enabled—한 레벨이라도 false면 비활성(AND, 안전측 실패)

같은 key가 양쪽에 있으면 구체 문서가 통째로 이깁니다(필드 단위 부분 병합 없음 — 결정적·예측가능). 구체 문서에만 있는 key는 추가됩니다.

sparse overlay 예시 — EQP001만 CPU 임계 95→85

// (*,*,*) 전역 문서는 그대로 두고, 아래 문서만 추가
{
  "scope": { "process": "PHOTO", "model": "MODEL_A", "eqpId": "EQP001" },
  "rules": [
    { "id": "cpu_crit", "interval_minutes": 5, "severity": "CRITICAL", "notify": "default",
      "when": [ { "fact": "cpu.max", "op": ">=", "value": 85 } ] }
  ]
  // measures/notify/그 외 rules는 전역 상속. cpu_crit가 참조하는 cpu measure도 전역에서 옴.
}

제약 (§6.4)

  • schedule/interval override는 process 레벨까지만. 스케줄러는 process 레벨 effective의 interval 집합으로 (process, interval) job을 등록합니다. model/eqp overlay가 process에 없는 새 interval을 도입하면 API가 422로 거부합니다(_check_interval_scope) — 안 그러면 그 rule이 스케줄되지 않아 silent lost breach. 임계/value override는 eqp 레벨까지 가능합니다.
  • 참조 무결성은 합성된 effective profile에서 검증합니다(§9).
❌ 절대 금지 특정 장비 하나를 바꾸려고 전역 설정을 통째로 복사하지 마세요(함정 P9). overlay엔 바꿀 measure/rule만 담고 나머지는 상속하게 둡니다 — 그래야 전역 변경이 자동 전파되고 드리프트가 없습니다.
소프트 tombstone 상속된 rule을 특정 scope에서만 끄려면, overlay에서 같은 id의 rule을 enabled:false로 다시 적습니다("통째로 교체"라 enabled까지 교체됨). 상속 항목을 완전히 삭제하는 기능은 아직 없고(추후), 이 enabled:false 뮤트가 그 역할을 대신합니다(§6 예시 ⑨).

§12 함정 9가지 (Pitfalls) SCHEMA §11

#함정올바른 사용
P1scope.eqpModel 카멜케이스Mongo 키는 eqpModel(snake eqp_model 아님). Python은 Scope 객체로만 다루기
P2scope.process ≠ measure.proc전자=적용범위/파티션, 후자=EARS 행 정체성
P3measureId.type 점 표기점 앞=measure id, 점 뒤=fact(=type). 둘 다 존재 강제
P4한 measure에 같은 type 중복금지(참조 모호). 다른 임계/창이 필요하면 measure 분리 또는 rule 분리
P5임계가 measure·rule 두 곳에spike_count.over=사건 경계(measure), rule.value=경보 기준(rule). 의도된 계층

P7category 필터 누락ES 쿼리에 EARS_CATEGORY.keyword term 필수(cpu/mem total_used_pct 혼입 방지)
P8활성 필터 누락EqpInfoRepository만 사용(decommissioned/미관리 장비 제외)
P9특정 장비 override 시 전역 통째 복사금지. overlay엔 바꿀 measure/rule만(나머지 상속). 전체 복사는 드리프트 유발

§13 Phase 1·2·3 구분 — 지금 동작 vs 후속 SCHEMA §13·§14 / PHASE_OF_FACT

스키마·검증은 16종 fact를 모두 받아들이지만, 실제 엔진이 평가하는 것은 현재 Phase 1뿐입니다. Phase 2/3 fact를 쓴 프로파일은 저장은 되지만 엔진이 skip하고 경고 로그를 남깁니다.

Phase상태fact비고
1구현완료(동작)max min avg last p50 p90 p95 p99 spike_countES 집계→rule 평가→이메일까지 end-to-end. 상태점검은 min/max로 흡수
2후속(skip+경고)duration delta growth_rate moving_avg trend zscoredate_histogram·extended_stats·top_hits 인프라 필요
3후속baseline_dev과거 N일 인덱스 별도 쿼리 필요

운영에 실제 시드되는 기본 프로파일(build_default_profile)은 Phase 1 fact만 포함한 축소판입니다. 아래 §14 예시는 전 Phase를 포함한 목표 설계입니다.
